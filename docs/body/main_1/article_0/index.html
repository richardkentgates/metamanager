What is Metamanager

Metadata governs how search engines, social platforms, AI crawlers, and editing software understand your content. Metamanager manages all of it — at both the file level and the web level.

Layer 1 — Media

File-Embedded Metadata

Standards written directly into asset files: EXIF, IPTC, XMP, ID3, QuickTime atoms, Vorbis comments. Managed by OS-level daemons using ExifTool and ffmpeg.

  • JPEG/PNG/WebP — EXIF, IPTC & XMP simultaneously
  • MP4/MOV/M4A — QuickTime atoms + XMP
  • MP3 — ID3 tags
  • OGG/FLAC — Vorbis comments
  • PDF — title, author, keywords, XMP write-back
  • GPS coordinates from camera EXIF to GeoCoordinates
  • Lossless compression: jpegtran, optipng, cwebp, ffmpeg
  • Bidirectional sync: file → WordPress and WordPress → file
Layer 2 — Web

Site & Page Metadata

Signals that search engines, social platforms, and structured data consumers read from your HTML. Controlled per-post, per-type, or site-wide from the Metamanager dashboard.

  • Title & description per post/page/term/user
  • Open Graph — image, video, audio, article, website types
  • Schema.org JSON-LD — 20+ types including LocalBusiness, Article, Event, Product
  • XML sitemaps — pages, media, video
  • HTML sitemap via [mm_sitemap] shortcode
  • Robots.txt control — per-type noindex + global rules
  • Async broken link checker
  • Business profile — Gutenberg block + Schema.org output
  • Author profiles with structured data

Why one plugin?

File metadata and web metadata are two sides of the same coin. A photo uploaded to WordPress carries GPS, creator, and copyright in its EXIF tags. Those same values should appear in the ImageObject Schema.org output. A business has a name and address in your WordPress settings — that same data should appear in LocalBusiness JSON-LD and in the contact card you embed on pages. Metamanager treats both layers as a single system with a single source of truth.

Architecture

Metamanager is designed around one principle: PHP coordinates, OS executes.

PrincipleIn practice
PHP coordinates, OS executesPHP never calls shell_exec() or any subprocess API. All media work is delegated to OS daemons via JSON job files. The daemon reads the file and calls jpegtran, ExifTool, or ffmpeg. PHP's role is to write the job and read the result.
No blocking in the request pathUpload hooks only write a small JSON file. They never wait for a daemon result. Processing happens via inotifywait + WP-Cron.
One source of truth per concernMM_DB owns schema and SQL. MM_Metadata owns field names and sync logic. MM_Job_Queue owns all job filesystem I/O. MM_Site_Settings owns all site-level metadata configuration.
Safe defaultsData is never deleted on uninstall unless explicitly opted in. Attribution fields (Creator, Copyright, Owner) are never set by bulk actions. Existing user values are never overwritten on import.
Minimal WordPress couplingClasses are plain PHP using WordPress APIs (wpdb, Options, Post Meta, WP-Cron, REST, WP-CLI) but have no hard dependency on plugin bootstrap order.

Quick Install

The install script detects your distro (apt / dnf / yum), installs all OS dependencies, copies the daemon scripts, installs systemd units, and activates the plugin. 

wget -qO- https://raw.githubusercontent.com/richardkentgates/metamanager/main/metamanager-install.sh | sudo bash

The script must run as root (or with sudo) to write systemd unit files and start services.

Manual Install

  1. Install OS dependencies for your distro: 
    # Ubuntu / Debian
sudo apt install -y libimage-exiftool-perl libjpeg-turbo-progs optipng webp ffmpeg inotify-tools jq

# AlmaLinux / RHEL / Rocky (requires EPEL + CRB)
sudo dnf install -y epel-release
sudo dnf config-manager --set-enabled crb
sudo dnf install -y perl-Image-ExifTool libjpeg-turbo-utils optipng libwebp-tools inotify-tools jq
# ffmpeg: RPM Fusion is required on RHEL-compatible distros
sudo dnf install -y https://mirrors.rpmfusion.org/free/el/rpmfusion-free-release-$(rpm -E %rhel).noarch.rpm
sudo dnf install -y ffmpeg
  2. Clone the plugin into your WordPress plugins directory: 
    cd /path/to/wordpress/wp-content/plugins
git clone https://github.com/richardkentgates/metamanager.git
  3. Copy systemd service files and enable daemons: 
    sudo cp metamanager/daemons/metamanager-*.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now metamanager-compress-daemon metamanager-meta-daemon
  4. Activate the plugin from the WordPress admin panel or via WP-CLI: 
    wp plugin activate metamanager

Post-Install Checklist

  • Confirm daemon status: Metamanager → Dashboard — green indicators mean both daemons are running.
  • Verify OS tools: the dashboard shows detection status for ExifTool, jpegtran, optipng, cwebp, and ffmpeg.
  • Configure site identity: Metamanager → Business Profile — set your site name, organization type, address, and social profiles.
  • Review title templates: Metamanager → Titles & Descriptions — set default title format and description fallback for each post type.
  • Enable sitemaps: Metamanager → Sitemaps — choose which post types and taxonomies to include.

Updating

Updates appear natively in Dashboard → Updates when a new release is published on GitHub. After updating, a notice will appear prompting you to restart the OS daemons:

sudo systemctl restart metamanager-compress-daemon metamanager-meta-daemon

This is required when daemon scripts change. The notice persists for 7 days and can be dismissed.

Lossless Compression

PHP cannot do lossless compression. Metamanager delegates all media work to purpose-built OS tools via background daemons — no re-encoding, no quality loss.

FormatToolMethod
JPEGjpegtranLossless optimization (no re-encoding). Strips unnecessary metadata if file already has embedded XMP/IPTC/EXIF preserved separately.
PNGoptipngLossless deflate optimization. File is only replaced if result is smaller.
WebPcwebp -losslessLossless re-encode. Original replaced only if output is smaller.
VideoffmpegContainer remux (not re-encode). Cleans malformed containers, writes metadata atoms. No video quality change.

Safety guarantee: A compressed file only replaces the original if it is strictly smaller in bytes. If the tool produces a larger output (rare for optipng/cwebp), the original is kept unchanged.

Bulk Compression

Run compression on all uncompressed uploaded media from Metamanager → Media Processing or via WP-CLI:

wp metamanager compress --all
wp metamanager compress --post-id=123

Bidirectional Sync

Sync works in both directions:

  • File → WordPress: On upload, ExifTool reads embedded tags and pre-populates empty WordPress fields. Existing user values are never overwritten.
  • WordPress → File: After you save a media item, the daemon reads the current WordPress values and writes them back into the file using ExifTool or ffmpeg.

Extended WordPress fields added by Metamanager: Creator, Copyright, Owner, Headline, Credit, Keywords, Date Created, Rating (0–5 stars), City, State/Province, Country, Publisher, Website URL. All fields are registered post meta, accessible via REST, and embedded by the daemon.

GPS & Location

GPS coordinates (latitude, longitude, altitude) embedded by camera hardware are automatically read from EXIF on upload and stored as _mm_gps_lat, _mm_gps_lng, _mm_gps_alt. When present, they are included in ImageObject JSON-LD output as a GeoCoordinates node — zero configuration required.

PDF Support

On PDF upload, Metamanager imports title, author, keywords, and creation date from the PDF document metadata. The daemon writes XMP back into the PDF. Schema.org output for PDFs uses DigitalDocument type.

Write-back Verification

After the daemon successfully writes metadata into a file, Metamanager queues a re-read job to verify the embedded values match what WordPress holds. Discrepancies are recorded in mm_verify_discrepancies post meta and shown as an amber warning on the attachment edit screen.

Page & Post Metadata

Per-page and per-post control over every metadata signal your site sends — title, description, canonical URL, and all structured data fields.

Metamanager adds a metadata panel to every post, page, custom post type edit screen, term editor, and user profile. Fields fall into three categories:

Title & Description

Override the default title and meta description for any individual post, page, or term. Default templates (configurable per post type) are used when no override is set. Variable tokens available: {title}, {site_name}, {sep}, {year}, {page}.

Canonical URL

Set a custom canonical URL on any post or page. Defaults to the WordPress permalink. Useful for content syndication and paginated posts.

Per-post robots

Set noindex, nofollow, noarchive, or nosnippet on individual posts without touching the global robots.txt configuration.

HTML Sitemap

Use [mm_sitemap] shortcode to embed a formatted HTML sitemap on any page. Supports post_types, taxonomies, and depth attributes.

[mm_sitemap post_types="post,page" depth="2"]

Robots.txt

Active sitemaps are automatically appended as Sitemap: directives to robots.txt via the robots_txt filter. Per-post-type noindex rules disallow crawling of selected types globally. Individual posts can be set noindex from the metadata panel.

Ping on Publish

Google and Bing receive a non-blocking sitemap ping whenever a post is published. Configurable in Metamanager → Sitemaps.

Link Management

An async broken link checker that runs in the background, plus a hygiene audit tool for managing link attributes across your content.

Broken Link Checker

Links discovered in your content are stored in a lightweight database table and checked asynchronously via wp_remote_head(). HTTP response codes are recorded. Links returning 4xx or 0 (timeout) are flagged as broken and surface in the Link Management dashboard.

  • Checks run in background batches — zero impact on page load
  • Per-link re-check button in the dashboard
  • Ignore toggle to suppress false positives
  • Optional email alerts when new broken links are found
  • Configurable check interval and timeout

Link Hygiene

Global rules for adding rel="nofollow", rel="noopener noreferrer", or target="_blank" to external links. Filter by domain pattern. Applied at render time via output buffer — no content modification stored.

Business Profile

A centralized organization profile that powers Schema.org output, the contact card block, local business structured data, and social link aggregation.

Profile Fields

Set once in Metamanager → Business Profile:

  • Organization name, type (Organization, LocalBusiness, NGO, etc.), description, logo URL
  • Address: street, city, state/region, postal code, country
  • Contact: phone, email, website
  • Social profiles: Facebook, Twitter/X, LinkedIn, Instagram, YouTube, Pinterest, TikTok
  • Business hours (per day), price range, area served
  • Geo coordinates (latitude, longitude)

Contact Card Block

The Business Contact Gutenberg block embeds a formatted contact card anywhere on your site. It pulls data from the business profile automatically. The metamanager/business-contact block renders name, address, phone, email, hours, and a map link.

Schema.org Output

The business profile generates an Organization or LocalBusiness JSON-LD block on every page, alongside the page-specific schema. Social profiles are included as sameAs links. When geo coordinates are set, a GeoCoordinates node is included.

Author Profiles

Extend each WordPress user's profile with structured data fields that power Person JSON-LD output on author archive pages and attributed posts.

Fields added to the user profile editor:

  • Job title, organization affiliation
  • Short biography (supplements the built-in WordPress bio)
  • Social profiles: Twitter/X, LinkedIn, GitHub, personal website
  • Profile image URL override

On author archive pages and posts where the author is shown, Metamanager emits a Person JSON-LD block with name, url, jobTitle, description, and sameAs links to social profiles.

Job Dashboard

Live view of the compression and metadata job queue, searchable history, daemon health indicators, and bulk action controls — all in one screen.

Job Queue

Every compression or metadata write operation creates a JSON job file in the appropriate queue directory. The daemon picks it up via inotifywait and processes it. The dashboard shows:

  • Currently queued and in-progress jobs
  • Searchable, paginated job history with status (completed / failed)
  • One-click retry for any failed job
  • Amber clock icon in the Media Library column for attachments with pending jobs

Daemon Health

Status indicators for both daemons (compress and meta) determined by PID file presence — no systemctl privilege required. Detection status for all OS tools (ExifTool, jpegtran, optipng, cwebp, ffmpeg) is shown inline.

Upload Notifications

The site admin receives a batched digest email after uploads (one email per 60-second window). Other WordPress users can opt in via their profile page. A configurable extra CC address is available. Failed sends surface as a dismissible admin notice with one-click retry.

IP Restriction

From Settings → REST API, enter a comma-separated list of allowed IP addresses or CIDR ranges. Requests from unlisted IPs receive a 403 before any capability check runs. Leave blank to allow all.

Disable all endpoints

A single toggle completely disables all Metamanager REST endpoints. This does not affect WordPress core REST routes.

Multisite

Metamanager is multisite-compatible. Network activation creates the metamanager_jobs database table and schedules WP-Cron on every existing site in the network. New site creation is handled via wp_initialize_site — no manual steps needed.

Each site maintains its own metadata settings, business profile, and link database. The OS daemons are shared across all sites (they watch a single job directory tree under wp-content/metamanager-jobs/).

Contributing

Metamanager is open source under GPL v3. Contributions of all kinds are welcome.

Branch structure

BranchPurpose
mainStable, tagged releases only. Protected. Receives merges from release/* and hotfix/*.
developIntegration branch. Default target for all feature PRs. Always deployable to a staging environment.
feature/*New features. Branch from develop, PR back into develop.
fix/*Non-urgent bug fixes. Branch from develop, PR into develop.
hotfix/*Urgent production fixes. Branch from main, merge into main AND develop.
release/*Release preparation (changelog, version bump). Branch from develop, merge into main AND develop. Tag on main.
docs/*Documentation-only changes. Branch from develop.

Pull requests

  • Target develop for features and fixes; target main only for hotfixes.
  • All PRs must pass PHPStan level 5 (run vendor/bin/phpstan analyse includes/ metamanager.php --level=5).
  • Follow the existing code style: WordPress coding standards, tabs for indentation, spaces around operators.
  • Add a CHANGELOG entry in the appropriate version section.
